# PRD: 회원(로그인) & 고객 데이터 관리 기반 (Membership & CRM Foundation)

> 상태: 제안 (v1) · 작성일: 2026-07-24
> 목표: 향후 광고·판매 기능 확장을 대비해, 회원 로그인 + 기초 CRM(고객 데이터 관리) 기반을 구축한다.

---

## 1. 결론 요약 (Feasibility)

**구현 가능하지만, 현재 사이트에 없는 계층(DB + 인증)을 신규로 도입해야 한다.**
BLOGSITE는 지금 완전 정적(SSG) 구조로, 콘텐츠는 GitHub 커밋으로 발행되고 별도 DB가
없다. 스튜디오 로그인(`ADMIN_PASSWORD`)은 운영자 1인용 비밀번호 체계일 뿐 회원 개념이
아니다. 따라서 회원가입/로그인/고객 데이터 관리는 **기존 파이프라인 위에 얹는 것이
아니라, 별도의 신규 레이어(DB + 인증 시스템)를 추가**하는 작업이다.

- 이 레이어는 기존 콘텐츠 발행·스튜디오·샵 결제 흐름과 **독립적으로 추가 가능** →
  개발 도중이든 완료 후든 현재 블로그 운영에 영향을 주지 않는다 (7장 참조).
- 예상 구현 기간: MVP(로그인 + 기본 고객 DB) **2~3주**, 타겟팅 준비까지 전체
  **6~8주**(파트타임 기준 캘린더로 2~3개월).
- 예상 추가 비용: DB **월 $0~25**(무료 티어로 시작 가능), 인증 자체는 무료.

## 2. 배경 및 문제

- 현재는 블로그(콘텐츠)와 편집샵(Stripe 게스트 결제)만 있고, "누가 방문·구매했는지"를
  앱이 전혀 기억하지 못한다. Stripe 대시보드에만 주문 기록이 남는다.
- 향후 광고·판매 기능을 고도화하려면 최소한 다음이 필요하다: 회원 식별(로그인),
  구매/행동 이력의 회원 귀속, 마케팅 동의 관리, 세그먼트 태깅.
- 이 네 가지는 모두 "회원이라는 개념이 DB에 존재해야" 가능하므로, 광고·판매 기능
  자체보다 먼저 이 기반 작업을 해두는 것이 재작업을 줄인다.

## 3. 목표 / 비목표

### 목표
- 이메일/비밀번호 + 소셜 로그인(카카오/네이버/구글)으로 방문자가 회원가입·로그인할 수
  있다.
- 회원 정보, 마케팅 수신 동의, 구매 이력(Stripe 연동)을 DB에 저장한다.
- 개인정보보호법·정보통신망법 기준에 맞는 동의 수집·철회 체계를 갖춘다.
- 이후 광고 타겟팅/세그먼트 캠페인에 바로 활용 가능한 데이터 구조(태그, 행동 이벤트)를
  마련한다.

### 비목표 (v1에서 하지 않음)
- 실제 광고 픽셀 연동(GA4, Meta Pixel 등) 및 캠페인 운영 — 데이터 구조만 준비, 연동은
  후속 PRD에서 다룬다.
- 마이페이지 고도화(포인트/등급/쿠폰 등 리텐션 기능) — 로그인 이후 검토.
- 스튜디오(운영자) 인증 체계 변경 — `ADMIN_PASSWORD` 방식은 그대로 유지한다(4장 참조).
- 결제 자체 로직 변경 — Stripe Checkout 흐름은 유지하고 "회원 귀속"만 추가한다.

## 4. 현재 아키텍처 vs 목표 아키텍처

| | 현재 | 목표(추가 후) |
|---|---|---|
| 콘텐츠 저장 | MDX 파일, GitHub 커밋 | **변경 없음** |
| 운영자 인증 | `ADMIN_PASSWORD` + HMAC 쿠키 (`lib/adminAuth.ts`) | **변경 없음** — 완전히 별도 네임스페이스 유지 |
| 결제 | Stripe Checkout, 게스트 전용 | Stripe Checkout 유지 + 로그인 시 회원에 귀속(선택) |
| 회원 데이터 | 없음 | **Supabase**(Postgres + 내장 Auth), tealdot 계정 내 별도 프로젝트(Free 티어) |
| 배포 | Vercel Hobby, DB 없이 정적+서버리스 | 동일 Vercel 프로젝트 + Supabase 프로젝트 연동(env var) |

핵심 설계 원칙: **기존 3개 시스템(콘텐츠/스튜디오/결제)을 건드리지 않고, 그 옆에 회원
시스템을 신규로 붙인다.** 스튜디오 로그인과 고객 로그인은 쿠키 이름, 세션 검증 로직,
UI 진입점(`/studio` vs `/login`)이 완전히 분리되어 서로 간섭하지 않는다.

**DB/인증 제공자 결정 (Phase 1)**: Supabase 선정. tealdot 사업자로 이미 운영 중인
quote(독서기록 앱)와 **같은 조직 계정, 별도 프로젝트**(Free 티어)로 분리한다 — 계정
재사용으로 학습·운영 비용을 아끼되, 두 서비스의 고객 데이터는 프로젝트 단위로 완전히
격리한다. Auth.js 대신 **Supabase 내장 Auth**를 사용해 스택을 하나 줄이고, RLS(Row
Level Security)로 "회원 본인 데이터만 접근 가능"을 DB 레벨에서 강제한다. 단, Supabase
Auth는 카카오는 네이티브 지원하지만 **네이버는 미지원**이라 Custom OAuth2 Provider로
직접 연결해야 한다(Free 플랜도 커스텀 프로바이더 3개까지 가능해 제약은 없음).

## 5. 데이터 모델 (초안)

`auth.users` / `auth.identities` / `auth.sessions`는 Supabase Auth가 자동 관리한다
(이메일/비밀번호, 카카오·네이버·구글 OAuth 식별자, 세션 토큰 포함) — 별도 테이블을
만들지 않는다. 앱이 직접 관리하는 건 Phase 0에서 확정한 항목뿐이다.

```
public.profiles (id = auth.users.id, FK, 1:1)
 ├─ name, phone
 ├─ marketing_consent: boolean, marketing_channels: text[] ("email"|"sms"|"kakao")
 ├─ created_at, updated_at
 └─ RLS: 본인 행만 select/update 가능

public.consent_log (법적 증빙용 동의 이력 — 수정이 아닌 append-only)
 ├─ user_id → auth.users
 ├─ type: "terms" | "privacy" | "marketing" | "age_over_14"
 ├─ granted: boolean, created_at
 └─ RLS: 본인 행만 select 가능, insert는 서버(서비스 롤)에서만

public.customer_links (Stripe 연동)
 ├─ user_id → auth.users (unique)
 ├─ stripe_customer_id
 └─ RLS: 본인 행만 select 가능, write는 서버(서비스 롤/웹훅)에서만
 └─ (주문 목록은 Stripe API에서 조회, 앱 DB에 중복 저장하지 않음)
```

- 원칙: **Stripe가 이미 잘 하는 것(결제/주문 원장)을 앱 DB에 복제하지 않는다.**
  `stripe_customer_id`로 연결만 해두고 조회 시 Stripe API를 사용한다. 데이터 정합성
  이슈와 PCI 관련 부담을 줄인다.
- `consent_log`를 append-only로 두는 이유: 개인정보보호법상 "언제 동의/철회했는지"
  증빙이 필요하며, `profiles.marketing_consent` 필드 하나만으로는 이력이 남지 않는다.
  만 14세 이상 확인도 별도 테이블 대신 `type: "age_over_14"` 행으로 기록해 스키마를
  늘리지 않는다.
- `auth.users` 신규 가입 시 `public.profiles` 행을 자동 생성하는 트리거
  (`handle_new_user()`)를 둔다 — 애플리케이션 코드가 매번 두 번 쓰기를 하지 않도록.

## 6. 단계별 로드맵

| Phase | 내용 | 산출물 | 기간 |
|---|---|---|---|
| **0. 요구사항/법적 설계** | 수집 항목 확정, 개인정보처리방침 개정안, 마케팅 동의 문구 | 문서 | 3~5일 |
| **1. DB 인프라 도입** | Supabase 프로젝트 생성(tealdot 계정, 별도 프로젝트), 스키마·RLS 마이그레이션 | DB + 스키마 | 3~5일 |
| **2. 인증 구현** | Supabase Auth 셋업, 이메일/비번 가입·로그인, 카카오(네이티브)·구글(네이티브)·네이버(Custom OAuth2) 연동 | `/login`, `/signup` | 1~2주 |
| **3. 고객 데이터 연동** | 회원 프로필 페이지, Stripe Customer 연결, 구매이력 조회 | 마이페이지 기초 | 1주 |
| **4. 동의관리 체계** | 쿠키 동의 배너, 마케팅 옵트인/아웃 UI, `ConsentLog` 기록 | 동의 UI | 3~5일 |
| **5. 세그먼트/타겟팅 기반** | 행동 이벤트 트래킹(조회/장바구니/구매), 태그 필드, 세그먼트 export 구조 | 이벤트 스키마 | 1~2주 |
| **6. QA·배포·모니터링** | 인증 흐름 테스트, 접근권한 점검, 프로덕션 배포 | 배포 | 3~5일 |

**MVP(Phase 0~3)**: 2~3주 · **전체(Phase 0~6)**: 6~8주(캘린더 기준 2~3개월)

## 7. 기존 운영에 미치는 영향 (중요)

**개발이 진행되는 동안에도, 완료된 이후에도 현재 블로그·스튜디오·샵 운영에는 영향이
없도록 설계한다.**

- **개발 중**: 별도 브랜치에서 작업하고, `main`에는 준비가 끝난 만큼만 병합한다. 신규
  라우트(`/login`, `/signup`, `/account`)와 신규 API(`/api/auth/*`)만 추가되며, 기존
  라우트(`/`, `/insight/*`, `/briefs`, `/shop/*`, `/studio`, `/api/admin/*`,
  `/api/checkout`)는 코드 변경이 없다.
- **환경변수**: `DATABASE_URL`, `AUTH_SECRET` 등 신규 변수만 **추가**되며, 기존 변수
  (`ADMIN_PASSWORD`, `GITHUB_TOKEN`, `STRIPE_SECRET_KEY` 등)는 그대로 둔다. 신규 변수가
  없으면 회원 기능은 비활성화되고 사이트는 지금처럼 정적으로 동작한다(스튜디오의
  "미설정 시 비활성화" 패턴과 동일하게 설계).
- **빌드/배포**: Next.js SSG로 빌드되는 콘텐츠 페이지는 DB 유무와 무관하게 계속
  빌드타임에 생성된다. 회원 관련 라우트만 동적(서버리스 함수)으로 추가되므로, 기존
  페이지의 로딩 속도나 캐싱 전략에 영향이 없다.
- **결제**: 게스트 체크아웃은 그대로 유지한다. 로그인 회원에게는 "구매이력 보기"가
  추가로 제공될 뿐, 결제 흐름 자체(성공/실패 처리)는 변경하지 않는다.
- **리스크**: 유일하게 주의할 부분은 DB 서비스가 다운되거나 설정이 잘못됐을 때
  `/login` 등 신규 라우트만 오류가 나야 하며, 이것이 다른 페이지에 전파되지 않도록
  인증 관련 코드는 반드시 격리된 모듈(`lib/auth.ts` 등)에서 실패를 흡수하도록 구현한다.

## 8. 법적·컴플라이언스 체크리스트

- 개인정보처리방침(`app/(site)/privacy/page.tsx`) 개정: 수집 항목, 위탁(Stripe/OAuth
  제공자), 보관기간, 파기 절차 명시.
- 회원가입 시 **이용약관 동의(필수)**와 **마케팅 수신 동의(선택)**를 분리하여 받는다
  (선택 동의 미체크가 가입을 막지 않아야 함 — 정보통신망법).
- 마케팅 동의 철회 경로(마이페이지 또는 수신거부 링크)를 반드시 제공한다.
- 만 14세 미만 회원 가입 시 법정대리인 동의 절차 필요 여부 검토(연령 확인 UI 포함
  여부는 Phase 0에서 결정).
- 소셜 로그인 제공자(카카오/네이버/구글)별 개발자 등록 및 서비스 약관 검토.
