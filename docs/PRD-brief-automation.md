# PRD: Briefs 자동화 파이프라인 (Auto-Brief)

> 상태: 제안 (v1) · 작성일: 2026-07-07
> 목표: 외부 기사/뉴스를 자동 수집 → LLM 요약 → Briefs 단신으로 발행하는 파이프라인 구축

---

## 1. 결론 요약 (Feasibility)

**구현 가능하다.** 현재 사이트 구조(파일 기반 MDX + GitHub 커밋 → Vercel 자동 재배포)는
자동화 파이프라인의 "발행" 단계가 이미 완성된 상태다. 스튜디오 CMS가 쓰는 것과 동일한
경로(저장소에 `.mdx` 커밋)를 봇이 대신 수행하면 되므로, **추가 서버·DB 없이
GitHub Actions 하나로 전체 파이프라인이 돌아간다.**

- 예상 운영 비용: **월 $1 미만** (LLM API 호출비만 발생, 나머지는 무료 티어)
- 예상 구현 규모: 스크립트 1개(~300줄) + 워크플로 1개 + 설정 파일 2개
- 핵심 리스크: 콘텐츠 품질(오요약) → **초안(draft) 발행 + 스튜디오 승인**으로 통제

## 2. 배경 및 문제

- Briefs는 "발 빠른 단신"이 정체성인데, 수동 운영 시 매일 뉴스 확인 → 요약 →
  발행의 반복 노동이 필요하다. 지속 가능성이 낮다.
- 반면 단신은 형식이 정형화되어 있어(2~4문장 + 태그 + 원문 링크) 자동화 적합도가
  아티클보다 훨씬 높다.

## 3. 목표 / 비목표

### 목표
- 지정한 소스에서 새 기사를 주기적으로 수집한다.
- LLM으로 **독자적인 한국어 요약**(2~4문장)을 생성한다.
- 기존 Briefs 포맷 그대로(`content/briefs/*.mdx`) 자동 발행한다.
- 원문 링크·출처를 항상 명시하고, 중복 발행을 방지한다.
- 운영자가 품질을 통제할 수 있는 승인 모드를 제공한다.

### 비목표 (v1에서 하지 않음)
- 아티클(Insights) 자동 생성 — 심층 글은 사람의 관점이 핵심이므로 제외
- 원문 전문 저장/재게시 — 저작권 정책상 금지 (7장 참조)
- 실시간(분 단위) 속보 — 하루 2~3회 배치로 충분
- 자체 크롤러로 임의 사이트 스크래핑 — RSS/공식 피드만 사용

## 4. 아키텍처 시나리오 비교

| | 시나리오 A (권장) | 시나리오 B | 시나리오 C |
|---|---|---|---|
| 실행 주체 | **GitHub Actions cron** | Vercel Cron + API Route | 외부 노코드(n8n/Make/Zapier) |
| 트리거 | 스케줄(cron) + 수동 실행 | Vercel 스케줄러 | 각 서비스 스케줄러 |
| 발행 방식 | 저장소에 직접 커밋 | GitHub API로 커밋 | Webhook → GitHub API |
| 비용 | 무료 (public repo 무제한, private 월 2,000분) | Hobby는 cron 2개/일 1회 제한 | 무료 티어 제한적, 유료화 가능성 |
| 장점 | 코드·설정·로그가 저장소에 일원화, 수동 재실행 쉬움 | 코드베이스 통합 | 코딩 최소화 |
| 단점 | YAML/스크립트 작성 필요 | 실행시간 제한(10s~60s), 크론 제약 | 외부 의존, 디버깅 어려움 |

**선정: 시나리오 A.** 이미 저장소가 발행의 단일 진실 공급원(SSOT)이므로, 수집·요약
로직도 같은 저장소의 Actions로 두는 것이 관리 포인트를 늘리지 않는 유일한 선택이다.
Vercel Hobby의 크론 제약(일 1회)과 함수 실행시간 제한도 회피한다.

## 5. 파이프라인 상세 설계

```
┌─ GitHub Actions (cron: 07:30, 12:30, 18:30 KST) ─────────────────┐
│                                                                    │
│  1. 수집   RSS/Atom 피드 파싱 (sources.yml에 등록된 소스만)        │
│      ↓     항목별: 제목, 링크, 발행시각, 피드 제공 요약            │
│  2. 중복   data/briefs-seen.json 과 대조 (URL 정규화 + 해시)       │
│      ↓     이미 처리한 항목 제거                                   │
│  3. 필터   키워드/카테고리 관심사 매칭, 발행 24h 이내만,           │
│      ↓     실행당 최대 N건(기본 3건)으로 제한                      │
│  4. 요약   Claude API (claude-haiku-4-5) 호출                      │
│      ↓     입력: 제목 + 피드 요약 (원문 전문 아님)                 │
│      ↓     출력: 한국어 단신 2~4문장 + 태그 2개 + 신뢰도 점수      │
│  5. 검증   길이(80~300자)·금칙어·태그 형식·신뢰도 임계값 체크      │
│      ↓     실패 항목은 스킵하고 로그만 남김                        │
│  6. 발행   content/briefs/YYYY-MM-DD-<slug>.mdx 생성               │
│      ↓     (운영 모드에 따라 draft: true 부여 — 8장)               │
│  7. 기록   briefs-seen.json 갱신 → 커밋 & 푸시                     │
│                                                                    │
└──── push → Vercel 자동 재배포 → 사이트 반영 (1~2분) ──────────────┘
```

### 생성되는 MDX 예시

```yaml
---
date: "2026-07-07T07:35:00+09:00"
tags: ["AI", "Tech News"]
link: "https://example-news.com/original-article"
source: "TechCrunch"          # 신규 필드: 출처 표기
automated: true               # 신규 필드: 자동 생성 표시
draft: true                   # 승인 모드일 때만
---
(LLM이 생성한 독자적 요약 2~4문장)
```

## 6. 구성 요소 스펙

### 6.1 소스 설정 — `automation/sources.yml`

```yaml
sources:
  - name: TechCrunch
    feed: https://techcrunch.com/feed/
    tags: [Tech News]
    lang: en                # 영→한 번역 요약
  - name: 지디넷코리아
    feed: https://zdnet.co.kr/news/news_xml.asp
    tags: [Korea, Tech News]
    lang: ko
filters:
  include_keywords: [AI, 반도체, 브랜드, 스타트업, 플랫폼, 커머스]
  max_items_per_run: 3
  max_age_hours: 24
```

- 소스 추가/삭제 = 이 파일 수정 후 커밋. 스튜디오에서 편집 가능하게 확장 여지 있음.
- **RSS/Atom 공식 피드 또는 공식 문서화된 공개 API만 허용.** 둘 다 아닌 소스(비공식
  스크래핑 등)는 등록 불가(7장 정책, 2026-07-24 개정).
- `type: huggingface` 소스(모델/데이터셋 메타데이터, 6.2 참조)는 `feed` 대신 `api`,
  `kind`(model|dataset), `newest`(선택, true면 최신순 + 24시간 필터 적용 / 기본 false면
  트렌딩순 + 기간 필터 미적용)를 사용한다.

### 6.2 요약 스크립트 — `automation/generate-briefs.mjs`

- Node 스크립트(의존성: `rss-parser`, `@anthropic-ai/sdk` 정도).
- LLM 프롬프트 원칙:
  - 입력은 **제목 + 피드가 제공하는 요약문**으로 제한 (전문 크롤링 안 함)
  - "사실 전달 + 한 줄 시사점" 구조, 기존 수동 브리프 톤 유지
  - 원문 문장 인용·번역 전재 금지, 재구성된 요약만 출력
  - JSON 스키마 강제 출력: `{ summary, tags, confidence }`
  - `confidence < 0.7`이면 발행 스킵
- 모델: `claude-haiku-4-5` (단신 요약에 충분, 비용 최소)

### 6.3 워크플로 — `.github/workflows/auto-brief.yml`

```yaml
on:
  schedule:
    - cron: "30 22,3,9 * * *"   # KST 07:30 / 12:30 / 18:30
  workflow_dispatch:            # 수동 실행 버튼
```

- Secrets: `ANTHROPIC_API_KEY` 1개만 필요 (커밋은 기본 `GITHUB_TOKEN` 사용)
- 커밋 메시지 규격: `bot: publish N briefs (YYYY-MM-DD HH:mm)`
- 동시 실행 방지: `concurrency` 그룹 지정

### 6.4 중복 방지 — `automation/briefs-seen.json`

- 처리한 기사 URL의 정규화 해시 목록 (최근 60일치만 유지, 자동 정리)
- 같은 사건을 여러 소스가 보도하는 경우: v1은 URL 기준만, v2에서 제목 유사도
  (임베딩 코사인) 기반 사건 단위 중복 제거 검토

## 7. 저작권·법적 가드레일 (필수 준수)

1. **공식 RSS/Atom 피드 또는 공식 문서화된 공개 API만 수집** (2026-07-24 개정) —
   피드 제공은 재배포가 아닌 "알림" 목적의 공개 신호이며, 전문 스크래핑 대비 법적
   리스크가 낮다. 공식 API도 동일한 성격(운영사가 문서화해 공개한 조회용 엔드포인트,
   인증 우회·비공개 API 역공학 아님)일 때만 같은 원칙으로 허용한다.
   예: Hugging Face Hub API(`huggingface.co/api/models`, `/api/datasets`) — HTML
   스크래핑이 아닌 공식 문서화 REST API.
2. **원문 전재 금지** — 저장·발행하는 것은 LLM이 재구성한 독자적 요약뿐. 원문
   문단의 복사·직역 전재를 하지 않는다 (프롬프트 + 검증 단계에서 이중 차단). API
   메타데이터(모델/데이터셋 카드)의 경우도 메타데이터에 없는 사실을 지어내지 않는다.
3. **출처 명시 의무화** — 모든 자동 브리프에 `source`(매체명) + `link`(원문 URL)
   필수. UI에서 "원문 보기"가 항상 노출된다.
4. **로봇 정책 존중** — robots.txt/ToS에서 봇 접근을 금지한 소스는 등록하지 않는다.
5. **삭제 요청 대응** — 출처 매체의 요청 시 해당 브리프를 즉시 삭제한다(스튜디오
   삭제 기능으로 처리 가능).
6. **소스 단위 장애 격리** — 소스(RSS든 API든) 하나가 실패해도 해당 소스만 스킵되고
   나머지 소스와 이번 실행 전체는 계속 진행된다(6.2 참조). 접근이 막히거나 API
   스키마가 바뀌어도 파이프라인 전체가 멈추지 않는다.

## 8. 운영 모드 (단계적 신뢰 구축)

| 모드 | 동작 | 적용 시점 |
|---|---|---|
| **Phase 1: 승인 모드** (기본) | 모든 자동 브리프를 `draft: true`로 발행 → 사이트 미노출 → 스튜디오 대시보드에서 "초안" 배지 확인 후 발행 버튼(= draft 해제 저장) | 출시 직후 2~4주 |
| **Phase 2: 신뢰 소스 자동 발행** | `sources.yml`에 `auto_publish: true` 플래그가 있는 소스만 즉시 발행, 나머지는 초안 | 품질 검증 후 |
| **Phase 3: 완전 자동** | 전 소스 즉시 발행 + 스튜디오는 사후 관리용 | 선택 사항 |

승인 UX는 기존 스튜디오를 그대로 활용한다(초안 목록·수정·삭제가 이미 구현되어
있음). 추가로 필요한 것은 "초안 → 발행" 원클릭 버튼 정도.

## 9. 품질 게이트 & 실패 처리

- **검증 실패**(길이/형식/신뢰도): 해당 항목만 스킵, 다음 실행에서 재시도 안 함
  (seen 처리) — 오발행보다 미발행이 낫다는 원칙
- **피드 다운**: 소스별 독립 처리, 한 소스 실패가 전체를 막지 않음
- **LLM API 장애**: 전체 중단 후 다음 크론에서 자연 재시도 (seen 미기록)
- **관측성**: Actions 실행 로그 + 커밋 히스토리가 곧 감사 기록. 실패 시 GitHub
  알림 메일 자동 발송
- **킬 스위치**: 워크플로 파일의 스케줄 주석 처리 또는 Actions 탭에서 disable

## 11. 구현 로드맵

| 단계 | 산출물 | 규모 |
|---|---|---|
| M1 | `sources.yml` + 수집/중복제거 스크립트 (요약 없이 dry-run 로그) | 0.5일 |
| M2 | LLM 요약 + 검증 + draft 발행, 워크플로 연결 | 0.5일 |
| M3 | 스튜디오 "초안 → 발행" 원클릭 버튼, BriefItem에 출처 표기 UI | 0.5일 |
| M4 | 2주 승인 모드 운영 → 품질 리포트 → Phase 2 전환 판단 | 운영 |

## 12. 리스크 및 완화

| 리스크 | 완화 |
|---|---|
| LLM 오요약/환각 | 피드 요약만 입력(전문 미사용) + 신뢰도 게이트 + 승인 모드 |
| 특정 소스 편중 | 실행당 소스별 최대 1건 제한 옵션 |
| 피드 포맷 변경 | 소스별 독립 파싱 + 실패 알림 |
| 자동 글 톤 이질감 | 기존 수동 브리프 3건을 few-shot 예시로 프롬프트에 포함 |
| API 키 유출 | GitHub Secrets 저장, 로그 마스킹 기본 제공 |
