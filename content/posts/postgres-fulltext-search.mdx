블로그에 검색을 붙이려 할 때 가장 먼저 떠오르는 선택지는 Algolia나 Elasticsearch 같은 전용 검색 엔진이다. 그러나 콘텐츠가 수천 건 이하라면, 이미 사용 중인 PostgreSQL이 충분히 훌륭한 검색 엔진이다.

## tsvector라는 자료구조

PostgreSQL의 풀텍스트 검색은 문서를 `tsvector`(정규화된 단어와 위치의 목록)로, 질의를 `tsquery`로 변환한 뒤 매칭한다. 핵심은 이 변환을 쿼리 시점이 아니라 저장 시점에 해두는 것이다.

```sql
ALTER TABLE posts ADD COLUMN fts_vector tsvector
  GENERATED ALWAYS AS (
    to_tsvector('simple', coalesce(title, '')) ||
    to_tsvector('simple', coalesce(summary, '')) ||
    to_tsvector('simple', coalesce(content_txt, ''))
  ) STORED;

CREATE INDEX posts_fts_idx ON posts USING gin(fts_vector);
```

Generated Column으로 선언하면 본문이 수정될 때마다 벡터가 자동 갱신된다. 애플리케이션 코드에서 동기화를 신경 쓸 필요가 없다.

## 다차원 필터의 조합

검색어 하나만으로는 부족하다. 실제 탐색은 카테고리(단일), 태그(다중), 키워드(텍스트)의 조합으로 일어난다.

> 좋은 검색 UX의 절반은 쿼리 성능이고, 나머지 절반은 상태를 URL에 남기는 습관이다.

태그 배열에는 PostgreSQL의 배열 포함 연산자 `@>`가 정확히 대응한다. `tags @> ARRAY['Branding','UX']`는 두 태그를 모두 가진 글만 남긴다. 각 축에 GIN 인덱스를 얹으면 세 조건이 결합되어도 인덱스 스캔으로 처리된다.

### 한글 검색의 함정

`to_tsvector('korean', ...)`을 쓰려면 한글 형태소 분석기가 필요하지만, Supabase 기본 인스턴스에는 내장되어 있지 않다. 현실적인 절충은 `simple` 사전으로 공백 단위 토큰화를 쓰되, `ILIKE` 폴백을 함께 두는 것이다. 제목과 요약처럼 짧은 필드에서는 이 조합으로도 체감 품질이 충분히 나온다.

## 검색 상태는 URL에 산다

`/blog?category=Design&tags=Branding,UX&q=minimal` — 필터 상태를 URL 쿼리 스트링에 매핑하면 검색 결과가 공유 가능한 자원이 된다. 뒤로 가기가 자연스럽게 동작하고, 서버 컴포넌트는 URL만 읽어 렌더링하면 된다. 상태 관리 라이브러리가 필요 없어지는 순간이다.

---

검색 엔진을 도입하기 전에 물어야 할 질문은 "검색이 필요한가"가 아니라 "지금 가진 데이터베이스가 그 검색을 못 하는가"다. 대부분의 경우, 답은 '할 수 있다'였다.
