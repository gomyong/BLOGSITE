React Server Components(RSC)를 처음 접하면 "서버에서 렌더링되는 컴포넌트"라는 표면적 정의에 머물기 쉽다. 그러나 RSC가 바꾸는 것은 렌더링 위치가 아니라 **데이터와 UI 사이의 거리**다.

## 클라이언트 중심 시대의 비용

SPA 시대의 프론트엔드는 "일단 빈 껍데기를 보내고, 클라이언트에서 데이터를 채운다"는 모델 위에 서 있었다. 이 모델의 비용은 명확하다.

- 데이터 페칭 워터폴 — 컴포넌트가 마운트되어야 요청이 시작된다.
- 번들 사이즈 — 데이터 변환 로직까지 전부 클라이언트로 배송된다.
- 로딩 상태의 폭발 — 모든 컴포넌트가 스피너를 가진다.

> 우리는 UI를 서버에서 클라이언트로 옮겨왔지만, 데이터는 여전히 서버에 있었다. RSC는 그 어긋남을 바로잡는다.

## RSC의 핵심은 '기본값의 역전'

Next.js App Router에서 모든 컴포넌트는 기본적으로 서버 컴포넌트다. 클라이언트 상호작용이 필요한 지점에만 `"use client"`를 선언한다. 이 기본값의 역전이 아키텍처 전체를 바꾼다.

### 서버 컴포넌트가 잘하는 일

파일 시스템을 읽고, 데이터베이스에 직접 쿼리하고, 무거운 파싱 라이브러리를 사용하는 일. 이 모든 것이 번들에 단 1바이트도 추가하지 않는다. 이 블로그의 글 목록 페이지가 정확히 이 패턴이다 — `fs.readFileSync`로 마크다운을 읽는 코드는 서버에만 존재한다.

### 클라이언트 컴포넌트가 잘하는 일

상태, 이벤트, 브라우저 API. 검색창의 디바운스 입력, 스크롤 진행률 바, 다크 모드 토글처럼 사용자의 손끝과 맞닿는 지점만 클라이언트로 내려보낸다.

## 경계 설계가 곧 아키텍처

RSC 이후의 설계 질문은 "이 컴포넌트를 어떻게 만들까"에서 "서버와 클라이언트의 경계를 어디에 긋을까"로 바뀐다. 경계가 낮은 곳(리프 컴포넌트)에 그어질수록 번들은 가벼워지고, 서버의 데이터 접근성은 최대로 유지된다.

---

기술의 성숙은 종종 기본값의 교체로 나타난다. RSC는 프론트엔드의 기본값을 클라이언트에서 서버로 되돌렸고, 우리는 이제야 웹의 원형에 더 가까운 아키텍처로 돌아가고 있다.
